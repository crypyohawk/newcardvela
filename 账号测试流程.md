## 扩容策略

### 渠道数量够用后只换 token，不加渠道

当端口/渠道数量达到目标（如 20 个）后，**不再新增端口和渠道**。
账号过期或额度耗尽时，只需：
1. 用户获取新 token（`npx copilot-api auth`）
2. AI 在该端口上执行「换绑操作」（更新 token → 重启 → 改渠道名）

这样 new-api 渠道结构保持稳定，所有请求路由不受影响。

## AI 操作流程

### 前置：获取 token（用户手动）

用户在服务器上执行：
```bash
npx copilot-api auth
```
按提示在浏览器中登录 GitHub Copilot 账号完成授权，然后执行：
```bash
cat /root/.local/share/copilot-api/github_token
```
把输出的 `ghu_xxx` token 和账号邮箱告诉 AI。

### 操作 A：换绑操作（在已有端口上切换/更新账号）

> 最常用操作。端口和渠道已存在，只替换 token。
> 适用：账号过期换新、额度耗尽换新、同账号 token 续期。

**AI 需要的信息**：端口号 PORT、新 token TOKEN、新邮箱 EMAIL（续期时邮箱不变）

```bash
# ===== 步骤 1：验证新 token =====
TOKEN="ghu_xxx"  # 用户提供的 token
curl -s -w "\nHTTP %{http_code}" https://api.github.com/copilot_internal/v2/token \
  -H "Authorization: token $TOKEN" | tail -3
# 必须返回 HTTP 200，否则停止！403 = Copilot 未订阅
curl -s https://api.github.com/user -H "Authorization: token $TOKEN" | grep '"login"'

# ===== 步骤 2：写入新 token 和启动脚本 =====
PORT=4143  # 目标端口
echo -n "$TOKEN" > /root/copilot-homes/$PORT/.local/share/copilot-api/github_token
cat > /root/copilot-homes/$PORT/start.sh << EOF
#!/bin/bash
export HOME=/root/copilot-homes/$PORT
exec /root/.npm/_npx/7d9acc9b18fee6ac/node_modules/.bin/copilot-api start --port $PORT --github-token $TOKEN
EOF
chmod +x /root/copilot-homes/$PORT/start.sh

# ===== 步骤 3：重启 PM2 实例 =====
# 进程名可能是 copilot-XXXX 或 cardvela-XXXX，先查再重启
pm2 list | grep $PORT
pm2 restart copilot-$PORT 2>/dev/null || pm2 restart cardvela-$PORT
pm2 save

# ===== 步骤 4：验证 =====
sleep 2
curl -s -o /dev/null -w "HTTP %{http_code}" http://127.0.0.1:$PORT/v1/models
# 期望: 200

# ===== 步骤 5：更新 new-api 渠道名称（换新账号时）=====
EMAIL="NewAccount@gmail.com"  # 新账号邮箱
CHANNEL_ID=4  # 该端口对应的渠道ID，先查：
# curl -s "http://127.0.0.1:3001/api/channel/?p=0&page_size=20" \
#   -H "Authorization: Bearer JyLH32oVwyGqUPUgNx8oqsvgaVItBc2F" \
```

## 架构说明

### HOME 隔离（为什么每个端口独立目录）

copilot-api **硬编码**数据目录为 `$HOME/.local/share/copilot-api/`。
多实例共用 HOME → 共用 token 文件 → 所有流量打到同一个 GitHub 账号 → 配额秒光。

每个端口独立的文件结构：
```
/root/copilot-homes/
├── ecosystem.config.js       # PM2 配置
├── 4141/
│   ├── start.sh              # export HOME=本目录 + exec copilot-api
│   └── .local/share/copilot-api/github_token
├── 4142/
│   ├── start.sh
│   └── .local/share/copilot-api/github_token
└── ...
```

### 关键 CLI 参数

- `--github-token`（别名 `-g`）：正确的 token 参数
- ~~`--token`~~：**不存在**，会被静默忽略！

## ⚠️ 禁止操作

1. **不要 `pkill -f copilot-api`** — 会产生孤儿进程占用端口，PM2 无法绑定 → 疯狂重启
2. **不要 `pm2 kill`** — 会杀死所有 PM2 管理的服务（包括 cardvela 主站、compat-proxy 等）
3. **只用 PM2 操作**：`pm2 restart/stop/delete/start`

紧急重建（万不得已）：
```bash
pm2 stop copilot-4141 copilot-4142 copilot-4143 copilot-4144 copilot-4145 copilot-4146 cardvela-4147
pm2 delete copilot-4141 copilot-4142 copilot-4143 copilot-4144 copilot-4145 copilot-4146 cardvela-4147
ss -tlnp | grep 414  # 确认端口释放
pm2 start /root/copilot-homes/ecosystem.config.js
pm2 save
```

## 故障排查记录

### 2026-04-10：所有实例共享同一 GitHub 账号

**现象**：Claude Code 返回 402 "You have no quota"
**根因**：6 实例共用 HOME → 共用 token → 一个账号扛全部流量 → 配额耗尽
**修复**：HOME 隔离方案（独立 HOME + `--github-token` 双重保障）

### 2026-04-10：孤儿进程导致端口冲突

**现象**：PM2 重启 100+ 次，`EADDRINUSE`
**根因**：`pkill` 杀进程后 PM2 重建产生多个 God Daemon，各自抢占端口
**修复**：杀死孤儿 daemon → 删除 `.pm2/` 目录 → 从 ecosystem 重建

### 2026-04-10：claude-haiku-4-5-20251001 未映射

**现象**：503 "No available channel"
**根因**：渠道只映射了 `claude-haiku-4-5`，没映射带日期后缀的变体
**修复**：所有渠道添加 `claude-haiku-4-5-20251001` → `claude-haiku-4.5`

### 2026-04-14：Copilot 订阅过期导致疯狂重启

**现象**：4145 重启 2399 次，4146 重启 5007 次
**根因**：GitHub 账号 Copilot Pro 过期，`copilot_internal/v2/token` 返回 403
**修复**：停止实例 → 禁用渠道 → ecosystem 加 `max_restarts: 5`
**教训**：操作前**必须先验证** token 的 Copilot 订阅状态（HTTP 200 才继续）

### 2026-04-14：new-api 创建渠道 panic

**现象**：POST /api/channel/ 返回 `nil pointer dereference`
**根因**：new-api (Calcium-Ion/new-api) 的 bug，无论 key 是否为空都 panic
**绕过**：通过 SQLite 直接 INSERT 渠道（复制已有渠道 + 改名字和地址），然后 `docker restart new-api`
