# New API 自建部署清单

更新时间：2026-04-06

目标：把 CardVela 当前依赖的第三方 new-api 网关替换为自建 new-api，且不动现有 CardVela、GSalary、copilot-api 4141-4145 主链路。

最终链路：
CardVela -> 自建 new-api -> copilot-api 4141-4145 -> GitHub Copilot

------------------------------------------------------------
一、执行原则
------------------------------------------------------------

1. 先并行部署，再切换，不直接覆盖现网。
2. 第一阶段自建 new-api 先跑在 3002，验证完再决定是否迁到 3001。
3. copilot-api 实例继续沿用现有 4141-4145，不重做。
4. CardVela 只改 NEW_API_* 环境变量，不改业务代码。
5. 如果切换失败，随时把 CardVela 环境变量切回旧网关并 pm2 重启即可回滚。

------------------------------------------------------------
二、当前约束
------------------------------------------------------------

1. CardVela 生产目录：/opt/cardvela
2. CardVela 运行方式：PM2
3. copilot-api 多实例管理脚本：/opt/cardvela/scripts/copilot-pool.sh
4. copilot-api 端口：4141-4145
5. CardVela 代码默认 new-api 地址：http://127.0.0.1:3001
6. CardVela 默认读取 new-api SQLite 路径：/home/ubuntu/new-api/data/one-api.db
7. new-api 官方仓库：https://github.com/QuantumNous/new-api
8. new-api 官方推荐方式：Docker Compose
9. Docker 镜像：calciumion/new-api:latest

------------------------------------------------------------
四、服务器可执行命令
------------------------------------------------------------

说明：以下命令默认在 Ubuntu 服务器执行。

### 0. 进入服务器后先做备份

```bash
cd /opt/cardvela
cp .env.production .env.production.bak.$(date +%F-%H%M%S)
pm2 save
pm2 status
```

### 1. 确认 Docker 环境

```bash
docker --version
docker compose version
```

如果这里失败，先安装 Docker，再继续后续步骤。

### 2. 部署自建 new-api 到 3002

```bash
cd /home/ubuntu
rm -rf new-api
git clone https://github.com/QuantumNous/new-api.git
cd new-api
cp .env.example .env
```

### 3. 修改 new-api 的 docker-compose 端口映射

目标：不要占用现网 3001，先临时映射到 3002。

先查看原配置：

```bash
cd /home/ubuntu/new-api
grep -n "3000" docker-compose.yml
grep -n "ports:" docker-compose.yml
```

把端口映射改成宿主机 3002 到容器 3000。

如果默认是 3000:3000，执行：

```bash
cd /home/ubuntu/new-api
sed -i 's/3000:3000/3002:3000/g' docker-compose.yml
```

如果默认是 3001:3000，执行：

```bash
cd /home/ubuntu/new-api
sed -i 's/3001:3000/3002:3000/g' docker-compose.yml
```

改完再次确认：

```bash
cd /home/ubuntu/new-api
grep -n "3002:3000" docker-compose.yml
```

### 4. 启动自建 new-api

```bash
cd /home/ubuntu/new-api
docker compose pull
docker compose up -d
docker compose ps
```

### 5. 检查 new-api 是否启动成功

```bash
curl -I http://127.0.0.1:3002
ss -lntp | grep 3002
docker compose logs --tail=100
```

浏览器访问：

http://服务器IP:3002

先完成管理员初始化登录。

### 6. 确认 copilot-api 实例仍正常

```bash
cd /opt/cardvela
bash scripts/copilot-pool.sh status
ss -lntp | grep 414
curl -s http://127.0.0.1:4141/v1/models | python3 -m json.tool | head
curl -s http://127.0.0.1:4142/v1/models | python3 -m json.tool | head
curl -s http://127.0.0.1:4143/v1/models | python3 -m json.tool | head
curl -s http://127.0.0.1:4144/v1/models | python3 -m json.tool | head
curl -s http://127.0.0.1:4145/v1/models | python3 -m json.tool | head
```

### 7. 在自建 new-api 后台添加 copilot 渠道

每个渠道都按下面规则配置：

1. 类型：OpenAI
2. 分组：copilot
3. Base URL：必须用宿主机地址，不要用 127.0.0.1

建议地址：

```text
http://172.17.0.1:4141
http://172.17.0.1:4142
http://172.17.0.1:4143
http://172.17.0.1:4144
http://172.17.0.1:4145
```

如果不确定宿主机 docker0 地址，执行：

```bash
ip addr show docker0 | grep 'inet '
```

### 8. 在 new-api 后台补齐分组设置

必须至少确认两项：

1. 分组倍率里有 copilot
2. 用户可选分组里有 copilot

建议值：

```text
copilot: 1
```

### 9. 在自建 new-api 后台创建管理员 token

创建后先保存这几个值：

```text
NEW_API_BASE_URL=http://127.0.0.1:3002
NEW_API_ADMIN_TOKEN=这里填你新生成的管理员 token
NEW_API_ADMIN_USER=1
NEW_API_SQLITE_PATH=/home/ubuntu/new-api/data/one-api.db
```

### 10. 在自建 new-api 上做一次手工调用验证

先在后台创建一个测试 token，然后执行：

```bash
curl -s http://127.0.0.1:3002/v1/models \
	-H "Authorization: Bearer 这里替换成测试 token" \
	| python3 -m json.tool | head -40
```

再执行一次对话测试：

```bash
curl -s http://127.0.0.1:3002/v1/chat/completions \
	-H "Authorization: Bearer 这里替换成测试 token" \
	-H "Content-Type: application/json" \
	-d '{"model":"gpt-4o","messages":[{"role":"user","content":"say hi"}],"max_tokens":10}' \
	| python3 -m json.tool
```

验证成功标准：

1. 有正常 JSON 返回
2. new-api 后台日志有记录
3. 命中 copilot 分组渠道

### 11. 切 CardVela 到自建 new-api

先改环境变量：

```bash
cd /opt/cardvela
grep -n "NEW_API_" .env.production
```

用编辑器修改：

```bash
cd /opt/cardvela
nano .env.production
```

至少改成：

```env
NEW_API_BASE_URL=http://127.0.0.1:3002
NEW_API_ADMIN_TOKEN=替换成自建 new-api 的管理员 token
NEW_API_ADMIN_USER=1
NEW_API_SQLITE_PATH=/home/ubuntu/new-api/data/one-api.db
```

### 12. 重新加载 CardVela 环境并重启 PM2

注意：PM2 下必须先把 .env.production 注入当前 shell，再启动或重启进程。

```bash
cd /opt/cardvela
npm install
npx prisma generate
npm run build
set -a
source .env.production
set +a
pm2 restart all
pm2 status
```

### 13. 验证 CardVela 已经连到自建 new-api

先看 PM2 环境里是否已注入：

```bash
pm2 env 0 | grep NEW_API
```

再检查 CardVela 服务日志：

```bash
pm2 logs --lines 100
```

### 14. 做完整业务验证

验证顺序：

```bash
cd /opt/cardvela
curl -s "http://127.0.0.1:3000/api/config" | python3 -m json.tool | head -40
curl -s "http://127.0.0.1:3000/api/cron/sync-usage?secret=这里替换成CRON_SECRET" | python3 -m json.tool
```

然后在前端后台继续验证：

1. 新建一个平台 API Key
2. 用该 Key 真实调一次模型
3. 去自建 new-api 后台确认 token 与日志增长
4. 触发 sync-usage
5. 确认 CardVela 用户余额和统计变化

------------------------------------------------------------
五、切正式端口 3001 的命令
------------------------------------------------------------

只有在 3002 稳定验证后才做。

### 方案 A：继续用 3002，不动 3001

优点：最稳，回滚最简单。

### 方案 B：把自建 new-api 切到 3001

```bash
cd /home/ubuntu/new-api
sed -i 's/3002:3000/3001:3000/g' docker-compose.yml
docker compose up -d
docker compose ps
ss -lntp | grep 3001
```

如果 CardVela 也要切回默认地址，再改：

```bash
cd /opt/cardvela
nano .env.production
```

改成：

```env
NEW_API_BASE_URL=http://127.0.0.1:3001
```

然后：

```bash
cd /opt/cardvela
set -a
source .env.production
set +a
pm2 restart all
```

------------------------------------------------------------
六、失败回滚命令
------------------------------------------------------------

如果切 CardVela 后发现异常，立刻回滚：

```bash
cd /opt/cardvela
cp .env.production.bak.你备份时的时间戳 .env.production
set -a
source .env.production
set +a
pm2 restart all
pm2 status
```

如果只想停掉自建 new-api：

```bash
cd /home/ubuntu/new-api
docker compose down
docker compose ps
```

------------------------------------------------------------
七、验收标准
------------------------------------------------------------

1. http://127.0.0.1:3002 可打开自建 new-api 后台
2. 自建 new-api 的 copilot 渠道全部可用
3. 测试 token 能在自建 new-api 上成功调用模型
4. CardVela 能成功创建 new-api token
5. 平台用户 Key 能通过自建 new-api 正常调用
6. sync-usage 能正常从自建 new-api 拉到日志并同步扣费
7. 出现问题时，能在 5 分钟内切回旧网关

------------------------------------------------------------
八、容易踩坑
------------------------------------------------------------

1. new-api 在 Docker 内时，渠道地址不能填 127.0.0.1:4141，必须填宿主机地址。
2. copilot 分组不是只建渠道就够，还要补系统设置里的分组倍率和用户可选分组。
3. PM2 不一定会自动读到 .env.production，重启前必须先 source 到当前 shell。
4. CardVela 创建 token 后如果拿不到完整 key，优先检查 NEW_API_SQLITE_PATH 是否正确。
5. 不要一上来就抢占 3001，先在 3002 完成验证。

### 日常运维命令

```bash
# 查看所有服务状态
pm2 status

# 查看 CardVela 日志
pm2 logs cardvela --lines 100

# 查看 new-api 容器日志
docker logs new-api --tail 100

# 重启 CardVela（改完 .env 后）
cd /opt/cardvela && set -a && source .env && set +a && pm2 restart cardvela

# 重启 new-api
docker restart new-api

# 手动触发用量同步
curl -s "http://127.0.0.1:3000/api/cron/sync-usage?secret=CRON_SECRET值"

# 查看 copilot 实例状态
cd /opt/cardvela && bash scripts/copilot-pool.sh status
```

### 回滚方案

如果自建 new-api 出问题，只需把 CardVela .env 中的 NEW_API_* 改回旧地址，
然后 source + pm2 restart 即可，5 分钟内可完成回滚。
